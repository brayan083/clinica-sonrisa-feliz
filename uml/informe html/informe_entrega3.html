Excepciones Personalizadas

1.1 Jerarquía de excepciones

Todas las excepciones del dominio extienden de ClinicaException, que a su vez extiende RuntimeException. Esto las convierte en excepciones unchecked: no obligan a declarar throws en la firma del método, pero pueden capturarse con un único catch (ClinicaException e) en la capa de presentación. Cada excepción incluye un código de error textual que permite identificarla programáticamente.

RuntimeException └ ClinicaException // clase base del proyecto, con código de error ├ PacienteNoEncontradoException // PACIENTE_NO_ENCONTRADO ├ OdontologoNoEncontradoException // ODONTOLOGO_NO_ENCONTRADO ├ TurnoNoEncontradoException // TURNO_NO_ENCONTRADO ├ TurnoYaReservadoException // TURNO_YA_RESERVADO ├ MatriculaDuplicadaException // MATRICULA_DUPLICADA ├ DniDuplicadoException // DNI_DUPLICADO ├ OperacionNoPermitidaException // OPERACION_NO_PERMITIDA └ DatoInvalidoException // DATO_INVALIDO └ FechaInvalidaException // hereda DATO_INVALIDO

1.2 Descripción y cuándo se lanza cada excepción

ExcepciónCuándo se lanzaCapa
PacienteNoEncontradoExceptionAl buscar un paciente por ID o DNI que no existe en el repositorio.Servicio
OdontologoNoEncontradoExceptionAl buscar un odontólogo por ID o matrícula que no existe.Servicio
TurnoNoEncontradoExceptionAl buscar un turno por ID que no existe en el repositorio.Servicio

1.3 Ejemplo: clase base ClinicaException

public abstract class ClinicaException extends RuntimeException {
    private final String codigo;

    public ClinicaException(String codigo, String mensaje) {
        super(mensaje);
        this.codigo = codigo;
    }

    public String getCodigo() { return codigo; }

    @Override
    public String toString() {
        return "[" + codigo + "] " + getMessage();
    }
}

Manejo Robusto de Errores

2.1 Estrategia por capa

Las excepciones se lanzan en la capa más cercana al dominio y se propagan hacia arriba hasta ser capturadas en la presentación, donde se muestran mensajes amigables.

🔧 Capa Modelo

La lógica de negocio intrínseca (disponibilidad de agenda, turno futuro) genera estados que el servicio detecta y convierte en excepciones.

⚙️ Capa Servicio

Valida precondiciones y lanza ClinicaException con mensajes descriptivos. No captura excepciones propias; las propaga.

🎮 Capa Controlador

Coordina múltiples servicios y lanza OperacionNoPermitidaException cuando la composición de reglas lo exige.

💻 Capa Presentación

Captura ClinicaException con un único catch y muestra el mensaje. Los bucles de entrada re-solicitan datos inválidos sin terminar el flujo.

2.2 try-catch-finally en operaciones de IO

PersistenciaCSV utiliza ambos estilos: try-with-resources (cierre automático) para la mayoría de operaciones, y try-catch-finally explícito en cargarPacientes() para demostrar el control manual del ciclo de vida del reader, garantizando el cierre incluso ante excepciones inesperadas.

// try-catch-finally explícito — cargarPacientes()
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader(PACIENTES_CSV));
    // ... procesamiento línea a línea ...
} catch (IOException e) {
    throw e; // propaga al llamador para que decida
} finally {
    if (br != null) {
        try { br.close(); } catch (IOException ignored) {}
    }
}

2.3 Validación inmediata en la UI

ConsolaUtils centraliza todos los métodos de entrada con validación y re-intento automático:

  • leerTexto() — rechaza cadenas vacías o con solo espacios.
  • leerEmail() — verifica que el texto contenga @.
  • leerFechaNoPassada() — re-solicita si la fecha es anterior a hoy.
  • leerIdExistente() — acepta un Consumer<Long>; si lanza excepción, repite el ciclo.

5
Persistencia

La clase PersistenciaCSV implementa la persistencia usando archivos de texto plano con separador ; para evitar conflictos con comas en los datos. Mantiene cuatro archivos:

📄 pacientes.csv

ID, nombre, apellido, email, DNI, fecha de ingreso y datos de domicilio (calle, número, localidad, provincia).

📄 odontologos.csv

ID, nombre, apellido, email, matrícula profesional.

📄 turnos.csv

ID, ID paciente, ID odontólogo, fecha, hora, estado, legajo recepcionista.

📄 recepcionistas.csv

ID, nombre, apellido, email, legajo. Se carga primero por dependencia referencial de los turnos.

Al cargar los archivos, los constructores CSV ajustan automáticamente el contador de IDs estático: if (id >= contadorId) contadorId = id + 1, garantizando IDs únicos sin un método resetContador() explícito.

Garantía de datos: los datos se persisten en cada operación de escritura. Un shutdown hook confirma el cierre limpio. No se pierden datos entre sesiones.
